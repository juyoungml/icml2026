Agents, Code, and Tool Use

An agent is more than a language model: prompts, tools, memory, environment state, and recovery logic jointly determine performance.

Start with intuition

The mental model

Treat the agent as a control loop: observe state, plan, choose a tool, inspect the result, update memory, and decide whether to continue or recover.

Inside this area

Agent evaluation; tool use; code generation and verification; software tasks; multi-agent planning, search, and coordination.

Common methods

Planning and reflection; tool routing; memory; execution feedback; multi-agent roles; verifiers; environment and benchmark design.

Technical core

A common research pipeline

  1. 1Observe the task and environment
  2. 2Plan and select a tool
  3. 3Execute and inspect feedback
  4. 4Recover, revise, or finish under cost and safety limits
Agent utilitytask success − cost − unsafe actions − unrecovered errors

Success alone can hide expensive search, brittle tools, or unacceptable actions.

Make it concrete

Worked example: academic figure agent

  1. Retrieve relevant figure references.
  2. Plan which concepts and relationships to show.
  3. Render a draft and critique readability and faithfulness.
  4. Revise, while measuring whether each agent component adds value.
Pause and predict

Does the environment prevent leakage and reward shortcut behavior?

Look for evidence tied to task success; pass rates; tool errors; cost and latency; recovery from failure; contamination; realistic environment coverage; safety boundaries. Do not overclaim

Evaluation

How progress is tested

Task success; pass rates; tool errors; cost and latency; recovery from failure; contamination; realistic environment coverage; safety boundaries.

Useful comparison

A base-model benchmark isolates model capability. An agent benchmark evaluates the combined model, scaffolding, tools, and environment.

Paper 2 · Abstract-based previewComparison case

$\tau^2$-Bench: Evaluating Conversational Agents in a Dual-Control Environment

Why selected: Dual-control agent evaluation broadens the core case and shows another important part of this area.

Problem
Most agent benchmarks let only the AI change the environment, unlike support tasks where users must also take actions.
Approach
Model a shared telecom environment where agent and user have tools, with generated tasks and a state-constrained user simulator.
Evidence
Dual-control task performance and ablations separating reasoning failures from communication and coordination errors.
Takeaway
Agent performance drops when success requires reliably guiding another actor through state-changing actions.
Caution
Simulator fidelity and telecom coverage limit how directly the result transfers to real users and other domains.

Paper 3 · Abstract-based previewComparison case

Agent0-VL: Exploring Self-Evolving Agent for Tool-Integrated Vision-Language Reasoning

Why selected: Tool-grounded self-improvement broadens the core case and shows another important part of this area.

Problem
Text-only self-evaluation can hallucinate when checking visual reasoning and cannot ground every correction in observable evidence.
Approach
Combine solver and verifier roles with tools, structured critique, self-repair, and reinforcement learning in one VLM.
Evidence
Chart, geometry, and visual-science reasoning evaluations against the underlying Qwen-VL model.
Takeaway
Tool-grounded self-verification is reported to improve the agent without external reward annotation.
Caution
Self-evaluation and test tools may share blind spots; independent verification and tool-failure tests remain necessary.
Metadata cues
RL / policy optimization; Reasoning / test-time compute; Agents / tool use; Graphs / geometry · vision/video; language/llm
